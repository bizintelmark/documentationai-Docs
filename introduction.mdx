## Overview

Droka OS Logistics (BIM-CARGO) is a web-based invoice management demo for cargo and warehouse operations. It focuses on managing customers, building invoices from tracked cargo items and weights, and enforcing role-based access for different warehouse staff responsibilities.

The current implementation runs as a single-page React app with an in-memory data model for customers, invoices, and staff. You use the main home screen to switch staff roles, create and view invoices, manage customer records, and inspect activity history for recent operations.

<Callout kind="info">
  The repo runs as a demo/prototype: all state lives in local in-memory React state and resets on page reload. There is no authentication; you select roles from a Staff dropdown in the header to simulate RBAC.
</Callout>

## Core concepts

### Customers

Customers represent cargo clients with an account and pricing configuration used when calculating invoice totals.

Typical fields in the UI include:

- **Account number** – a unique identifier for the customer.
- **Name and contact details** – who the cargo belongs to and how to reach them.
- **Pricing type** – standard vs. VIP-tier pricing, which controls how weights are billed.
- **Custom price** – an optional per-kilogram override for VIP customers.

The customer form and listing are implemented in:

- `client/src/components/CustomerForm.tsx`
- `client/src/components/CustomerList.tsx`
- `client/src/components/CustomerLookup.tsx`

The pricing type drives how invoices compute totals. For VIP customers, the UI prefers a customer-specific `customPrice` over the general price.

```ts
const pricePerKg =
  selectedCustomer?.pricingType === "vip"
    ? selectedCustomer.customPrice || generalPrice
    : generalPrice;
```

### Invoices

Invoices combine one customer, tracking-based items, and a pricing model into a billable document. The UI assigns each invoice a compact, human-readable invoice number and tracks its lifecycle with statuses.

The invoice orchestration and views live in:

- `client/src/components/InvoiceForm.tsx`
- `client/src/components/InvoiceListView.tsx`
- `client/src/components/InvoicePreview.tsx`

#### Invoice number format

Invoice numbers follow a predictable pattern composed of a prefix, the current year, and a zero-padded sequence:

```ts
const invoiceNumber = `INV-${new Date().getFullYear()}-${String(
  invoices.length + 1
).padStart(3, "0")}`;
// Example: "INV-2024-003"
```

In the current demo, the sequence is derived from the length of the in-memory invoice array, so numbers reset when the page reloads.

#### Invoice status

Each invoice carries a status flag of `unpaid`, `paid`, or `cancelled` that controls how it appears in the list and preview components. Status changes are reflected in the activity history so you can see when an invoice was created or updated and by which staff member (based on the active role in the header).

### Tracking items

Tracking items model the cargo you bill for on an invoice. Each item typically includes:

- **Tracking reference** – the shipment or tracking ID.
- **Dimensions** – length, width, and height fields.
- **Weight** – the billable weight for pricing calculations.

The invoice total is calculated as the sum of each item's weight multiplied by the derived `pricePerKg`:

```ts
const total = items.reduce((sum, item) => {
  const weight = parseFloat(item.weight) || 0;
  return sum + weight * pricePerKg;
}, 0);
```

Because all data lives in local state, updating items immediately recomputes the total in the UI.

### Staff roles and RBAC

The home screen includes a Staff selector in the header that lets you switch between predefined staff identities and roles. Role-based access control is implemented purely in the UI: different roles enable or disable specific actions or panels.

Roles are defined as a string union type in `client/src/pages/home.tsx`:

```ts
export type StaffRole = "admin" | "manager" | "warehouse";
```

The demo seeds a small staff list:

```ts
const [staff] = useState<Staff[]>([
  { id: "1", name: "Jennifer", role: "admin" },
  { id: "2", name: "Carlos", role: "manager" },
  { id: "3", name: "Miguel", role: "warehouse" },
]);
```

You switch between staff via a dropdown, and the UI updates available actions based on the selected role rather than on any server-side authentication.

## Activity history

Activity history provides a lightweight audit trail of changes within the current browser session. Each time you create or update a customer or invoice, the app records a structured entry showing:

- What operation occurred (for example, invoice created, customer updated).
- Which entity it affected.
- Which staff member performed the action (based on the active role).
- When it happened (timestamp within the current session).

The history view is implemented in `client/src/components/RecordHistory.tsx` and is reset when the page reloads, consistent with the in-memory-only data model.

## How it works in the repo

The BIM-CARGO repo is organized as a React + TypeScript frontend with backend scaffolding for an eventual API layer.

- `client/src/pages/home.tsx` orchestrates the main screen, holds the in-memory demo state (customers, invoices, staff), and wires together all major components.
- UI modules:
  - `CustomerForm.tsx`, `CustomerList.tsx`, `CustomerLookup.tsx` handle customer CRUD and selection.
  - `InvoiceForm.tsx` manages invoice composition, including items and totals.
  - `InvoiceListView.tsx` shows existing invoices with filtering and status views.
  - `InvoicePreview.tsx` renders a printable-style invoice view.
  - `RecordHistory.tsx` renders the in-session activity log.
- Backend scaffolding:
  - `server/` contains an Express + Drizzle setup ready to back the UI with real persistence and APIs.
  - `shared/` holds schema and type definitions that can be shared between client and server.

In the current state, the UI uses only local React state and does not call the Express server. Turning the demo into a production app primarily involves wiring the home page and forms to real endpoints backed by the `server/` and `shared/` modules.
