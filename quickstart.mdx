## Prerequisites

<Callout kind="info">
Install and verify:

- Node.js (LTS recommended)
- npm (bundled with most Node.js installs)
- A modern browser (Chrome, Edge, or Firefox)

This demo uses in-memory data. When you restart the dev server or refresh hard during certain flows, the seeded state may reset.
</Callout>

## Run Droka OS Logistics locally

Start the backend server and client so you can use the customers and invoices UI.

<Steps>
  <Step title="Install dependencies" icon="download">
    
```bash
npm install
```

Your terminal finishes with all packages installed and no fatal errors.
  </Step>

  <Step title="Run in development mode" icon="play-circle">
    
```bash
npm run dev
```

The `dev` script starts the server with development settings:

```ts
// package.json → scripts (excerpt)
"dev": "NODE_ENV=development tsx server/index-dev.ts"
```

The server logs a line similar to:

```text
serving on port 5000
```

Then open your browser to:

```text
http://localhost:5000
```

If you set a `PORT` environment variable, the app listens on that port instead of `5000`.
  </Step>

  <Step title="Build and run in production mode (optional)" icon="package">
Use this only when you want to test the built assets.

```bash
# Build client and server bundle
npm run build

# Start production server
npm run start
```

Relevant scripts:

```json
"build": "vite build && esbuild server/index-prod.ts --platform=node --packages=external --bundle --format=esm --outfile=dist/index.js",
"start": "NODE_ENV=production node dist/index.js"
```

The production server uses the same port logic as development:

```ts
const port = parseInt(process.env.PORT || "5000", 10);
server.listen({
  port,
  host: "0.0.0.0",
  reusePort: true,
}, () => {
  log(`serving on port ${port}`);
});
```
  </Step>
</Steps>

<Callout kind="info">
If `PORT` is not set, Droka OS Logistics listens on `http://localhost:5000`. If that port is busy or you prefer a custom port, export `PORT=3000` (or another value) before running `npm run dev`.
</Callout>

## Understand roles and access

Droka OS Logistics does not have authentication or login in this demo. Instead, the UI simulates role-based access control using a Staff dropdown.

- Use the **Staff** selector (data-testid `select-current-staff`) in the header to switch between seeded staff identities.
- Each identity has a role:
  - **Admin**: full access (customers, invoices, activity).
  - **Manager**: invoice operations, but limited customer management.
  - **Warehouse**: operational view with restricted write actions.

Actions appear or disappear based on the selected role. If you do not see a button or menu item mentioned in this guide, switch to an Admin or Manager identity.

## Complete your first workflow

Follow this path once to get a feel for the main customer and invoice flows.

<Steps>
  <Step title="Open the app and pick a staff role" icon="monitor">
Open `http://localhost:5000` in your browser.

Locate the **Staff** dropdown in the top bar and switch to an **Admin** identity. The label may include both the staff member name and role.

Selecting Admin ensures you can create customers, create invoices, and review activity history in one session.
  </Step>

  <Step title="Create a customer (Admin only)" icon="users">
Go to the **Customers** section from the main navigation.

- Click **New Customer**.
- Enter basic details such as the customer name and contact information.
- Choose a **pricing type** for the customer (for example, a per-kilogram rate configuration used when calculating invoice totals).

Save the customer. You should see the new customer listed, and it becomes available when creating invoices.
  </Step>

  <Step title="Create an invoice for that customer" icon="file-text">
Switch to a role that can create invoices (**Admin** or **Manager**).

Navigate to the **Invoices** section and click **New Invoice**:

- Select the customer you created in the previous step.
- Confirm or adjust the customer pricing type if the UI exposes it.
- Add one or more tracking items:
  - Provide tracking details (for example, route or consignment reference).
  - Enter a **weight** value for each item.

The invoice form calculates the total using code similar to:

```ts
const total = items.reduce((sum, item) => {
  const weight = parseFloat(item.weight) || 0;
  return sum + weight * pricePerKg;
}, 0);
```

The total updates as you add or edit items. Save the invoice when you are satisfied with the line items.
  </Step>

  <Step title="Mark the invoice paid or cancelled and preview it" icon="check-circle">
After saving, you land on the invoice details page or see the invoice listed in the Invoices table.

From there:

- Use the status controls to **mark the invoice as Paid or Cancelled**.
- Open the **Preview** or **View** option to see the invoice as an output document.

Invoice numbers follow a predictable format, for example:

```ts
const invoiceNumber = `INV-${new Date().getFullYear()}-${String(invoices.length + 1).padStart(3, "0")}`;
```

This pattern uses the current year and a zero-padded sequence number, which helps when scanning multiple invoices during testing.
  </Step>

  <Step title="Review activity history (Admin)" icon="bar-chart">
Switch back to an **Admin** identity if needed.

Open the **Activity** or **History** section (naming may vary slightly in the UI). You see a timeline of actions such as:

- Customer created or updated
- Invoice created
- Invoice status changed to Paid or Cancelled

Confirm that your recent customer and invoice actions appear in this history. This is useful for validating that role-based UI flows are wired correctly end to end.
  </Step>
</Steps>

## Troubleshooting

<Callout kind="info">
Most quickstart issues relate to ports, environment variables, or role selection in the Staff dropdown.
</Callout>

<ExpandableGroup>
  <Expandable title="Page does not load on http://localhost:5000" default-open="false">

- Make sure the dev server is running with `npm run dev`.
- Check the terminal logs for `serving on port 5000` or another port.
- If you see a different port in the log (for example, `serving on port 3000`), open that port in your browser instead.
- If you want a specific port, set `PORT` before starting:

```bash
export PORT=3000
npm run dev
```

  </Expandable>

  <Expandable title="Actions are missing (cannot create customer or invoice)" default-open="false">

- Verify which staff role is selected in the **Staff** dropdown.
- Switch to an **Admin** identity when you need full access, including customer creation and activity history.
- Use a **Manager** identity to focus on invoice operations with limited configuration powers.
- The **Warehouse** role intentionally hides many write actions; use it only to test restricted views.

  </Expandable>

  <Expandable title="Data disappears after restart or reload" default-open="false">

- The app uses demo and in-memory state for many flows.
- If you stop and restart `npm run dev`, previously created customers and invoices may reset.
- When testing a particular scenario, keep the server running and avoid full hard reloads unless you expect to re-seed data.

  </Expandable>
</ExpandableGroup>
